## 1. 系统总体目标

### 1.1 业务目标

将当前依赖 Excel 手工分析的模式，升级为一个可持续运行的本地经营分析系统。

系统重点解决以下经营问题：

- 哪些行政区、哪些合伙人经营表现更好或更差
- 新合伙人开城后完成得怎么样
- 哪些合伙人单量出现大幅波动
- 单个城市每天、每小时的完成与取消情况如何
- 新骑手、新商家是否形成有效供给
- 总部补贴与合伙人补贴分别产生了什么效果

### 1.2 技术目标

- 自动读取订单明细、骑手名单、商家名单、合伙人名单
- 自动导入数据库并完成清洗、建模、汇总
- 自动生成本地网页看板
- 支持全国行政区分析和单个合伙人城市分析
- 支持后续按本文档重新生成程序

## 2. 技术选型

### 2.1 技术栈总览

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 编程语言 | Python 3.10+ | 数据处理生态成熟，开发效率高 |
| 数据库 | PostgreSQL 15+ | 关系型数据库，支持复杂查询、窗口函数、分区表 |
| Web 框架 | FastAPI | 异步高性能，自带 API 文档（Swagger） |
| 前端框架 | Vue 3 + ECharts | 交互式看板，图表丰富 |
| 数据处理 | Pandas + SQLAlchemy | 数据清洗与 ORM 操作 |
| 定时任务 | APScheduler | 数据导入与汇总的定时调度 |
| 日志 | Python logging + Loguru | 结构化日志，按级别分文件输出 |
| 配置管理 | PyYAML + pydantic | YAML 配置文件，带类型校验 |

### 2.2 选型理由

**为什么选 PostgreSQL 而非 SQLite？**

- SQLite 单文件锁机制不适合多进程并发
- PostgreSQL 支持窗口函数、CTE、分区表，复杂汇总查询性能更好
- 未来可扩展为远程部署，多用户同时访问

**为什么选 FastAPI 而非 Flask？**

- 原生支持异步，I/O 密集型查询性能更优
- 自带 Swagger UI，无需额外维护 API 文档
- Pydantic 模型校验，减少前后端联调问题

**为什么选 Vue 3 + ECharts？**

- ECharts 对热力图、折线图、柱状图支持完善
- Vue 3 响应式系统适合筛选联动场景
- 可打包为静态文件，直接用 FastAPI 托管

## 4. 主键与字段映射规则

> **关键规则**：本章节定义了所有表之间的关联关系，是系统准确性的基础，重建程序时必须严格遵守。

### 4.1 订单明细主字段

订单明细表至少使用以下字段：

- `订单编号`（主键）
- `合伙人ID`
- `商家ID`
- `用户ID`
- `配送员ID`
- `订单状态`
- `添加时间`
- `支付时间`
- `取消时间`
- `完成时间`
- `总部优惠金额`
- `营销优惠劵ID`
- `优惠金额`

### 4.2 骑手字段映射

来源文件：`帮手信息.xlsx`

| 源字段 | 目标字段 | 数据类型 | 说明 |
|--------|---------|---------|------|
| 帮手ID | rider_id | STRING | 主键，必须与订单明细的配送员ID一致 |
| 帮手姓名 | rider_name | STRING | |
| 入职时间 | hire_date | DATE | 用于判断新骑手 |
| 状态 | rider_status | STRING | |
| 所属合伙人 | rider_partner_id | STRING | 必须是合伙人ID而非名称 |
| 区域 | rider_region | STRING | |

**关键规则：**

- 主键关联：`帮手ID = 订单明细.配送员ID`
- 新骑手判断：只按 `入职时间` 判断
- 合伙人关联：所属合伙人字段必须是合伙人表的 ID，而非文本名称

> **如果源文件只有合伙人名称怎么办？**
> 需增加预处理步骤：通过合伙人名称匹配到 ID。无法匹配的记录写入 `ods_unmapped_riders` 表。

### 4.3 商家字段映射

来源文件：`商户信息.xlsx`

| 源字段 | 目标字段 | 数据类型 | 说明 |
|--------|---------|---------|------|
| 商家ID | merchant_id | STRING | 主键 |
| 商家名称 | merchant_name | STRING | |
| 所属合伙人 | merchant_partner_id | STRING | 必须是合伙人ID而非名称 |
| 所属区域 | merchant_region | STRING | |
| 注册时间 | merchant_register_date | DATE | 用于判断新商家 |
| 状态 | merchant_status | STRING | |

**关键规则：**

- 新商家判断：只按 `注册时间` 判断
- 合伙人关联：同骑手规则，必须用 ID 关联

### 4.4 合伙人字段映射

来源文件：`2026.4.1合伙人信息.xlsx`

| 源字段 | 目标字段 | 数据类型 | 说明 |
|--------|---------|---------|------|
| ID | partner_id | STRING | 主键，必须与订单明细的合伙人ID一致 |
| 合伙人公司名 | partner_name | STRING | |
| 成立时间 | partner_open_date | DATE | 用于判断新合伙人 |
| 合伙人区域 | partner_region_raw | STRING | 需拆分为省/市/区县 |
| 状态 | partner_status | STRING | |

**关键规则：**

- 主键关联：`ID = 订单明细.合伙人ID`
- 开城时间：`成立时间` 作为开城时间
- 行政区拆分：见下方详细规则

### 4.5 行政区拆分标准化规则

**输入示例：**

```
四川省-成都市-武侯区
北京市-北京市-朝阳区
新疆维吾尔自治区乌鲁木齐市天山区
```

**拆分规则：**

1. 分隔符优先级：优先按 `-` 分隔，其次 `/`，最后空格
2. 直辖市处理：拆分后只有2段时，自动补全省名 = 市名
   - 示例：`上海市-黄浦区` → `上海市 / 上海市 / 黄浦区`
3. 标准化映射：使用国家标准行政区码表（GB/T 2260-2007）进行映射
4. 异常处理：无法识别的行政区记录到 `ods_invalid_region` 表
5. 输出字段：`province`（省）、`city`（市）、`district`（区县）

## 5. 核心业务口径

> **重要提醒**：本章节定义的业务口径是所有指标计算的基础，任何变更必须经过业务团队确认。

### 5.1 有效订单定义

**公式：**

```
有效订单 = 完成订单 + 支付后取消时长 > 5分钟的订单
```

**判定规则：**

- **完成订单**：订单状态为已送达
- **支付后取消订单**：
  - 存在支付时间 AND 存在取消时间
  - 取消时间 - 支付时间 > 5分钟（注意是大于，正好5分钟不计入）
- **不计入有效订单**：
  - 未支付取消
  - 支付后 5 分钟内取消（含正好5分钟）

### 5.2 新主体定义

| 主体类型 | 判断依据 | 说明 |
|---------|---------|------|
| 新骑手 | 入职时间 | 按骑手名单 `入职时间` 判断，入职后 N 天内视为新骑手（N 默认30天，可配置） |
| 新商家 | 注册时间 | 按商家名单 `注册时间` 判断，注册后 N 天内视为新商家（N 默认30天，可配置） |
| 新合伙人 | 成立时间/开城时间 | 按合伙人名单 `成立时间` 判断，开城后 N 天内视为新合伙人（N 默认90天，可配置） |

> **说明**：新主体的"新"有观察窗口概念。入职/注册/成立超过观察窗口后，该主体不再被视为"新"。观察窗口可在 `config.yaml` 中配置。

### 5.3 补贴归属规则

| 字段组合 | 补贴归属 |
|---------|---------|
| 总部优惠金额 > 0 且 营销优惠劵ID 为空 | 总部补贴 |
| 总部优惠金额 > 0 且 营销优惠劵ID 不为空 | 合伙人补贴 |
| 优惠金额 > 0 | 合伙人补贴 |

> **注意**：一笔订单可能同时产生总部补贴和合伙人补贴（前者来自总部优惠金额，后者来自优惠金额），这是预期行为。

### 5.4 商家主体口径

商家保留三套分析口径：

| 口径名称 | 定义 | 用途 |
|---------|------|------|
| 商家ID口径 | 按商家ID统计 | 标准商家分析 |
| 用户ID口径 | 按用户ID统计 | 识别个人用户 vs 商家用户 |
| 商家型用户口径 | 在筛选范围内，用户ID累计完成订单量 >= 阈值 | 识别"伪装成用户的商家" |

**商家型用户阈值规则：**

- 默认阈值：20单
- 时间窗口：
  - 筛选范围 < 7天：按7天滚动窗口计算
  - 筛选范围 >= 7天：按实际筛选范围计算
- 可在 `config.yaml` 中调整

### 5.5 时间归属规则

**日汇总统一口径：**

- 按 `order_date`（从 `添加时间` 提取日期）归属
- 跨天订单标记：新增字段 `is_cross_day_order`
  - 完成时间日期 != 添加时间日期 → 标记为 1

**小时汇总口径：**

- 按 `添加时间` 的小时归属

**完成率计算：**

- 分母：添加时间 当天的订单
- 分子：实际完成的订单（可能跨天完成）

**特殊情况处理：**

| 情况 | 处理方式 |
|------|---------|
| 时间为空 | `order_date` 设为 NULL，单独统计"时间缺失订单数" |
| 时间格式错误 | 尝试智能解析（如 `2026/4/8` → `2026-04-08`），失败则记录异常 |

### 5.6 活跃主体定义

| 主体类型 | 活跃判定规则 |
|---------|------------|
| 活跃合伙人 | 在选定时间范围内，至少有1笔订单（含所有状态） |
| 活跃商家 | 在选定时间范围内，至少有1笔完成订单 |
| 活跃骑手 | 在选定时间范围内，至少有1笔完成订单 |

## 6. 数据库设计

### 6.1 数据分层架构

```
ODS（原始层）→ DIM（维度层）→ DWD（明细层）→ ADS（汇总层）
                                         ↓
                                    DQC（数据质量层）
```

### 6.2 原始层 ODS

| 表名 | 用途 |
|------|------|
| ods_file_registry | 文件导入注册表（文件名、哈希、导入时间、编码格式） |
| ods_order_detail_raw | 订单明细原始数据 |
| ods_rider_roster_raw | 骑手名单原始数据 |
| ods_merchant_roster_raw | 商家名单原始数据 |
| ods_partner_roster_raw | 合伙人名单原始数据 |
| ods_unmapped_riders | 无法关联合伙人的骑手记录 |
| ods_invalid_region | 无法识别的行政区记录 |

### 6.3 维度层 DIM

| 表名 | 用途 | 主键 |
|------|------|------|
| dim_rider | 骑手维度表 | rider_id |
| dim_merchant | 商家维度表 | merchant_id |
| dim_partner | 合伙人维度表 | partner_id |
| dim_calendar | 日历维度表 | date |
| dim_partner_change_log | 合伙人变更日志 | change_id |

**dim_calendar 字段说明：**

| 字段 | 说明 |
|------|------|
| date | 日期 |
| is_weekend | 是否周末 |
| is_holiday | 是否法定节假日 |
| is_campaign_day | 是否大促日（双11、618等） |
| campaign_name | 活动名称 |

用途：节假日、大促日波动预警豁免；支持同比分析（匹配相同星期几）。

**dim_partner_change_log 字段说明：**

| 字段 | 说明 |
|------|------|
| change_date | 变更日期 |
| old_partner_id | 原合伙人ID |
| new_partner_id | 新合伙人ID |
| change_type | 变更类型：rename / merge / split |
| note | 备注 |

用途：记录合伙人改名、合并、拆分，用于历史数据追溯。

**维度表更新策略（SCD Type 2）：**

- 当骑手/商家/合伙人信息变更时，新增一条记录并标记生效时间范围
- 查询时按订单日期匹配生效的维度记录
- 保留完整变更历史，支持回溯分析

### 6.4 明细层 DWD

表名：`dwd_order_detail`

**派生字段：**

| 字段名 | 说明 |
|--------|------|
| order_date | 订单日期（从添加时间提取） |
| order_hour | 订单小时（从添加时间提取） |
| is_paid | 是否已支付 |
| is_completed | 是否已完成 |
| is_cancelled | 是否已取消 |
| pay_cancel_minutes | 支付后取消分钟数 |
| is_valid_order | 是否为有效订单 |
| is_cross_day_order | 是否跨天订单 |
| hq_subsidy_amount | 总部补贴金额 |
| partner_subsidy_amount | 合伙人补贴金额 |
| is_new_rider_order | 是否新骑手订单 |
| is_new_merchant_order | 是否新商家订单 |
| is_new_partner_order | 是否新合伙人订单 |

> **关于 `merchant_like_user_flag`**：商家型用户判定依赖筛选范围，不适合在 DWD（明细层）静态生成。此字段移到 ADS 层查询时动态计算。

**索引设计：**

```sql
PRIMARY KEY (订单编号)
INDEX idx_partner_date (partner_id, order_date)
INDEX idx_merchant_date (merchant_id, order_date)
INDEX idx_rider_date (rider_id, order_date)
INDEX idx_region_date (province, city, order_date)
INDEX idx_order_status (order_status, is_valid_order)
```

| dqc_data_quality_log | 每次导入的数据质量指标 |
| dqc_abnormal_orders | 异常订单明细 |

### 6.7 数据分区策略

订单表按月分区（当数据量超过100万条时启用）：

```sql
CREATE TABLE dwd_order_detail_202604 PARTITION OF dwd_order_detail
FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');
```

查询限制：前端单次查询最多跨3个月。

### 6.8 数据生命周期管理

| 数据层 | 保留时长 | 归档策略 |
|--------|---------|---------|
| ODS 原始层 | 3个月 | 压缩为 Parquet 格式归档 |
| DWD 明细层 | 12个月 | 超期数据删除 |
| ADS 汇总层 | 永久保留 | 体积小，无需归档 |
| 小时汇总 | 90天 | 超期归档到 `_archive` 表 |

**归档流程：**

1. 每月1号自动触发
2. 导出 → 压缩 → 归档 → 删除源数据
3. 归档数据保留在 `archive/` 目录

## 7. 数据处理流程

### 7.1 文件导入流程

程序启动导入时执行以下步骤：

1. 扫描 `orders/`、`riders/`、`merchants/`、`partners/` 四个目录
2. 识别全部 `xls/xlsx/csv` 文件
3. 根据文件名、文件大小、SHA256 哈希进行去重
4. 编码自动检测：
   - 优先尝试 UTF-8 解码
   - 失败后尝试 GBK 解码
   - 仍失败则使用 `chardet` 库自动检测
   - 检测结果写入 `ods_file_registry.encoding` 字段
5. 将原始数据写入 ODS 表
6. 将导入结果写入日志表和日志文件

**字段名容错规则：**

| 规则 | 示例 |
|------|------|
| 支持中英文混合匹配 | 订单编号 = 订单号 = order_id = OrderID |
| 不区分大小写 | OrderID = orderid |
| 模糊匹配 | Levenshtein 距离 <= 2 |
| 无法匹配的列 | 在导入报告中列出，供人工确认 |

**事务控制：**

- 按文件事务（推荐）：每个文件独立事务，单文件失败不影响其他文件
- 导入状态机：`pending → importing → success / failed / partial_success`

**失败重试机制：**

- 自动重试3次，间隔5秒
- 3次失败后标记为 `failed`，记录详细错误
- 断点续传：程序重启后跳过已成功文件

**进度实时反馈：**

```json
{
  "total_files": 120,
  "processed_files": 45,
  "current_file": "订单明细_2026-04-01.xlsx",
  "current_progress": 0.68,
  "estimated_seconds_remaining": 180,
  "errors": []
}
```

### 7.2 数据清洗流程

清洗时执行以下步骤：

1. 标准化时间字段（统一格式 `YYYY-MM-DD HH:MM:SS`）
2. 统一空值、`-`、`0` 处理
3. 重复字段重命名（如源文件有两个"状态"列）
4. 金额字段转数值
5. ID 字段统一为字符串
6. 行政区字段拆分为 `省 / 市 / 区县`
7. 关联骑手、商家、合伙人维表

### 7.3 业务计算流程

业务计算时执行以下步骤：

1. 计算支付后取消分钟数（`pay_cancel_minutes`）
2. 判断是否为有效订单（`is_valid_order`）
3. 计算总部补贴金额（`hq_subsidy_amount`）
4. 计算合伙人补贴金额（`partner_subsidy_amount`）
5. 判断是否为新骑手订单（`is_new_rider_order`）
6. 判断是否为新商家订单（`is_new_merchant_order`）
7. 判断是否为新合伙人城市订单（`is_new_partner_order`）
8. 判断是否为跨天订单（`is_cross_day_order`）

### 7.4 汇总输出流程

汇总时执行以下步骤：

1. 生成全国行政区日汇总（`ads_admin_day_metrics`）
2. 生成行政区-合伙人汇总（`ads_admin_partner_metrics`）
3. 生成合伙人城市日汇总（`ads_partner_day_metrics`）
4. 生成合伙人城市小时汇总（`ads_partner_hour_metrics`）
5. 生成骑手日汇总（`ads_partner_rider_day_metrics`）
6. 生成商家日汇总（`ads_partner_merchant_day_metrics`）
7. 生成商家型用户汇总（`ads_partner_merchant_like_user_metrics`）

## 8. 数据质量管理

### 8.1 导入质量校验

每次文件导入后，自动执行以下校验并记录到 `dqc_data_quality_log`：

| 校验项 | 校验内容 | 告警条件 |
|--------|---------|---------|
| 行数校验 | 源文件行数 vs 入库行数 | 差异 > 0 |
| 编码校验 | 是否使用了降级编码（GBK/chardet） | 使用降级编码时记录 |
| 必填字段校验 | 订单编号、订单状态等必填字段 | 空值率 > 1% |
| 时间格式校验 | 添加时间、支付时间等时间字段 | 异常率 > 1% |
| 金额校验 | 优惠金额等数值字段 | 非数值率 > 1% |
| 维表关联率 | 订单关联合伙人/骑手/商家的比例 | 关联失败率 > 5% |

### 8.2 异常订单检测

以下情况记录到 `dqc_abnormal_orders` 表：

| 异常类型 | 判定规则 | 处理方式 |
|---------|---------|---------|
| 时间倒挂 | 取消时间 < 支付时间 | 标记异常，按正常处理 |
| 时间倒挂 | 完成时间 < 添加时间 | 标记异常，按正常处理 |
| 金额异常 | 优惠金额 > 订单金额 | 标记异常，人工复核 |
| 关联缺失 | 合伙人ID在维表中不存在 | 写入未映射表，按正常处理 |
| 重复订单 | 订单编号重复 | 保留最新记录，标记重复 |

### 8.3 汇总数据一致性校验

每次汇总完成后，执行以下校验：

- 各行政区完成单量之和 = 全国总完成单量
- 各合伙人日汇总完成单量之和 = 该日全国完成单量
- 小时汇总完成单量之和 = 当日完成单量
- 校验结果写入日志，不一致时告警

### 10.2 合伙人波动预警

**预警识别规则：**

根据城市日均单量分档设定阈值：

| 城市规模 | 日均单量 | 变化量阈值 | 变化幅度阈值 |
|---------|---------|-----------|------------|
| 大型城市 | >= 500单 | >= 100单 | >= 20% |
| 中型城市 | 200-499单 | >= 50单 | >= 25% |
| 小型城市 | < 200单 | >= 30单 | >= 30% |

**预警优先级：**

| 优先级 | 条件 |
|--------|------|
| 紧急 | 连续3天下滑 且 累计下滑 > 50% |
| 警告 | 单日下滑达阈值 |
| 关注 | 单日上涨达阈值 |

**智能归因（辅助信息）：**

同时展示以下变化指标，辅助定位原因：

- 新商家变化
- 新骑手变化
- 活跃商家变化
- 活跃骑手变化
- 取消率变化
- 补贴变化

示例输出：`成都合伙人订单下滑32%，主要原因：活跃骑手减少15人(-20%)，新商家仅新增2家`

**节假日豁免规则：**

- 节假日、大促日的波动不触发告警
- 但仍记录到日志，供事后分析
- 基于 `dim_calendar` 表判断

## 附录A. 核心规则清单

> **重要提醒**：后续重建程序时必须保留以下规则，任何变更需经过业务团队确认。

### A.1 主键关联规则

- `帮手ID = 订单明细.配送员ID`
- `商家ID = 订单明细.商家ID`
- `合伙人信息.ID = 订单明细.合伙人ID`
- 骑手/商家的"所属合伙人"字段必须是合伙人ID而非名称

### A.2 新主体定义规则

- 新骑手：只按 `入职时间` 判断，观察窗口默认30天
- 新商家：只按 `注册时间` 判断，观察窗口默认30天
- 新合伙人：只按 `成立时间/开城时间` 判断，观察窗口默认90天

### A.3 有效订单定义规则

- 有效订单 = 完成订单 + 支付后取消时长 > 5分钟的订单
- 注意：正好5分钟不计入

### A.4 补贴归属规则

- `总部优惠金额` 的归属由 `营销优惠劵ID` 决定：
  - 营销优惠劵ID 为空 → 总部补贴
  - 营销优惠劵ID 不为空 → 合伙人补贴
- `优惠金额` 永远记为合伙人补贴
- 一笔订单可能同时产生总部补贴和合伙人补贴

### A.5 商家主体口径规则

- 商家同时保留三套口径：`商家ID / 用户ID / 商家型用户`
- 商家型用户阈值默认20单，可配置

### A.6 时间归属规则

- 日汇总按 `添加时间` 的日期归属
- 小时汇总按 `添加时间` 的小时归属
- 跨天订单需单独标记

### A.7 行政区拆分规则

- 分隔符优先级：`-` > `/` > 空格
- 直辖市自动补全：省名 = 市名
- 使用国家标准行政区码表映射

### A.8 波动预警规则

- 分级阈值：大/中/小城市不同标准
- 节假日、大促日豁免告警
