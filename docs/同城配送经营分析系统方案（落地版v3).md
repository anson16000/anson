## 1. 系统总体目标

### 1.1 业务目标

将当前依赖 Excel 手工分析的模式，升级为一个可持续运行的本地经营分析系统。

系统重点解决以下经营问题：

- 哪些行政区、哪些合伙人经营表现更好或更差
- 新合伙人开城后完成得怎么样
- 哪些合伙人单量出现大幅波动
- 单个城市每天、每小时的完成与取消情况如何
- 新骑手、新商家是否形成有效供给
- 总部补贴与合伙人补贴分别产生了什么效果
- 哪些城市补贴投入高但经营效率没有同步提升
- 哪些城市存在明显的供给不足或高峰时段取消异常

### 1.2 技术目标

- 自动读取订单明细、骑手名单、商家名单、合伙人名单
- 自动导入数据库并完成清洗、建模、汇总
- 自动生成本地网页看板
- 支持全国行政区分析和单个合伙人城市分析
- 支持核心经营指标按统一口径反复复用
- 支持后续按本文档重新生成程序

### 1.3 建设目标定位

本项目定位为：

> **轻量级经营分析系统，而不是重型数据平台。**

第一阶段聚焦“跑通主流程、保证核心口径正确、提供可用分析页面”，不追求一步到位建设全量数仓平台能力。

## 2. 建设原则与范围边界

### 2.1 建设原则

本方案遵循以下原则：

1. **先落地，后扩展**：先完成可用 MVP，再逐步增强
2. **先统一口径，再做展示**：所有页面指标必须来自统一计算逻辑
3. **先保证准确性，再优化复杂度**：核心指标错误比缺少高级功能风险更大
4. **避免过度设计**：第一阶段不引入过重的平台化能力
5. **方便重建与移交**：后续即使更换开发人员，也能按本文档重建系统

### 2.2 第一阶段必须实现的能力

```text
数据导入 → 数据清洗 → 明细建模（DWD）→ 汇总生成（ADS）→ 看板展示
```

包括：

- 文件自动导入
- 基础字段映射与标准化
- 核心业务口径计算
- 两类核心汇总表
- 两个核心页面
- 基础日志与异常记录

### 2.3 第一阶段明确不做的内容

以下内容不纳入第一阶段交付范围：

- 维度表 SCD Type 2 历史版本管理
- 全量复杂数据质量平台
- 完整权限系统
- 复杂 API 开放平台
- 数据分区与归档体系
- AI 驱动的自动诊断与预测
- 多租户、多环境发布体系

### 2.4 第二阶段可扩展方向

在第一阶段稳定运行后，可按需扩展：

- 预警系统增强
- 数据质量体系完善
- 权限与脱敏控制
- API 对接外部系统
- 历史归档与性能优化
- 更丰富的商家、骑手、补贴专题分析

## 3. 技术选型

### 3.1 技术栈总览

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 编程语言 | Python 3.10+ | 数据处理生态成熟，开发效率高 |
| 数据库 | PostgreSQL 15+ | 支持聚合、索引、复杂查询，适合本地分析系统 |
| Web 框架 | FastAPI | 接口开发快，前后端联调方便 |
| 前端框架 | Vue 3 + ECharts | 页面开发效率高，图表展示能力强 |
| 数据处理 | Pandas + SQLAlchemy | 文件读取、清洗、入库便利 |
| 定时任务 | APScheduler | 第一阶段满足定时导入需求 |
| 日志 | Python logging | 轻量、可控，满足初期日志要求 |
| 配置管理 | YAML + pydantic | 配置清晰，支持校验 |

### 3.2 选型原则说明

**为什么选 PostgreSQL 而不是 SQLite？**

- SQLite 不适合多用户访问和中等规模查询
- PostgreSQL 支持更完整的聚合与索引能力
- 后续扩展时不需要更换数据库

**为什么保留 FastAPI？**

- 开发效率高
- 适合本地部署和内部使用
- 后续如要补 API，对接成本低

**为什么前端使用 Vue 3 + ECharts？**

- 适合做看板与筛选联动
- 热力图、趋势图、柱状图实现简单
- 开发成本较低，后期维护方便

### 3.3 第一阶段技术约束

为保证 2 周左右可落地，第一阶段增加以下技术约束：

- 不引入 Airflow、Kafka 等重型中间件
- 不做分布式调度
- 不做实时流式计算
- 不做复杂消息通知体系
- 不做多环境 CI/CD 体系

## 5. 主键与字段映射规则

> **关键规则**：本章节定义了所有表之间的关联关系，是系统准确性的基础，重建程序时必须严格遵守。

### 5.1 订单明细主字段

订单明细表至少使用以下字段：

- `订单编号`（主键）
- `合伙人ID`
- `商家ID`
- `用户ID`
- `配送员ID`
- `订单状态`
- `添加时间`
- `支付时间`
- `取消时间`
- `完成时间`
- `总部优惠金额`
- `营销优惠劵ID`
- `优惠金额`

### 5.2 骑手字段映射

来源文件：`帮手信息.xlsx`

| 源字段 | 目标字段 | 数据类型 | 说明 |
|--------|---------|---------|------|
| 帮手ID | rider_id | STRING | 主键，必须与订单明细的配送员ID一致 |
| 帮手姓名 | rider_name | STRING | |
| 入职时间 | hire_date | DATE | 用于判断新骑手 |
| 状态 | rider_status | STRING | |
| 所属合伙人 | rider_partner_id | STRING | 必须与合伙人ID关联 |
| 区域 | rider_region | STRING | |

**关键规则：**

- 主键关联：`帮手ID = 订单明细.配送员ID`
- 新骑手判断：只按 `入职时间` 判断
- 合伙人关联：所属合伙人字段必须优先使用 ID，而非文本名称

> 如果源文件只有合伙人名称，则需增加预处理映射；无法映射的记录写入异常表。

### 5.3 商家字段映射

来源文件：`商户信息.xlsx`

| 源字段 | 目标字段 | 数据类型 | 说明 |
|--------|---------|---------|------|
| 商家ID | merchant_id | STRING | 主键 |
| 商家名称 | merchant_name | STRING | |
| 所属合伙人 | merchant_partner_id | STRING | 必须与合伙人ID关联 |
| 所属区域 | merchant_region | STRING | |
| 注册时间 | merchant_register_date | DATE | 用于判断新商家 |
| 状态 | merchant_status | STRING | |

**关键规则：**

- 新商家判断：只按 `注册时间` 判断
- 合伙人关联：同骑手规则，优先使用 ID 关联

### 5.4 合伙人字段映射

来源文件：`2026.4.1合伙人信息.xlsx`

| 源字段 | 目标字段 | 数据类型 | 说明 |
|--------|---------|---------|------|
| ID | partner_id | STRING | 主键，必须与订单明细的合伙人ID一致 |
| 合伙人公司名 | partner_name | STRING | |
| 成立时间 | partner_open_date | DATE | 用于判断新合伙人 |
| 合伙人区域 | partner_region_raw | STRING | 需拆分为省/市/区县 |
| 状态 | partner_status | STRING | |

**关键规则：**

- 主键关联：`ID = 订单明细.合伙人ID`
- 开城时间：`成立时间` 作为开城时间
- 行政区拆分：用于全国行政区分析

### 5.5 行政区拆分规则

第一阶段采用轻量规则：

1. 优先按 `-` 分隔
2. 其次按 `/` 分隔
3. 最后按空格分隔
4. 直辖市只有2段时，补全省名 = 市名
5. 无法识别的记录写入异常表

**输出字段：**

- `province`
- `city`
- `district`

> 第一阶段不强制引入完整国家行政区码表，但建议预留标准化映射接口，便于第二阶段增强。

## 6. 核心业务口径

> **重要提醒**：本章节定义的业务口径是所有指标计算的基础，任何变更必须经过业务团队确认。

### 6.1 有效订单定义

**公式：**

```text
有效订单 = 完成订单 + 支付后取消时长 > 5分钟的订单
```

**判定规则：**

- **完成订单**：订单状态为已送达，或系统定义的完成状态
- **支付后取消订单**：
  - 存在支付时间
  - 存在取消时间
  - `取消时间 - 支付时间 > 5分钟`
- **不计入有效订单**：
  - 未支付取消
  - 支付后 5 分钟内取消
  - 正好 5 分钟取消

### 6.2 新主体定义

| 主体类型 | 判断依据 | 默认观察窗口 | 说明 |
|---------|---------|-------------|------|
| 新骑手 | 入职时间 | 30天 | 入职后 N 天内视为新骑手 |
| 新商家 | 注册时间 | 30天 | 注册后 N 天内视为新商家 |
| 新合伙人 | 成立时间/开城时间 | 90天 | 开城后 N 天内视为新合伙人 |

### 6.3 补贴归属规则

| 字段组合 | 补贴归属 |
|---------|---------|
| 总部优惠金额 > 0 且 营销优惠劵ID 为空 | 总部补贴 |
| 总部优惠金额 > 0 且 营销优惠劵ID 不为空 | 合伙人补贴 |
| 优惠金额 > 0 | 合伙人补贴 |

**说明：**

- 一笔订单可能同时产生总部补贴和合伙人补贴
- `总部优惠金额` 是否归属总部或合伙人，取决于营销优惠劵ID是否为空
- `优惠金额` 永远归属合伙人补贴

### 6.4 商家主体口径

商家保留三套分析口径：

| 口径名称 | 定义 | 用途 |
|---------|------|------|
| 商家ID口径 | 按商家ID统计 | 标准商家分析 |
| 用户ID口径 | 按用户ID统计 | 辅助识别主体偏差 |
| 商家型用户口径 | 用户ID累计完成订单量达到阈值 | 识别高频用户 |

**商家型用户阈值规则：**

- 默认阈值：20单
- 第一阶段仅在单城市分析页支持阈值展示
- 第一阶段不将该口径作为主汇总表静态字段写死

### 6.5 时间归属规则

**日汇总口径：**

- 按 `添加时间` 提取 `order_date`
- 所有日指标统一按 `order_date` 汇总

**小时汇总口径：**

- 按 `添加时间` 提取 `order_hour`

**完成率计算口径：**

- 分母：当日添加订单数
- 分子：最终完成订单数

**特殊处理：**

| 情况 | 处理方式 |
|------|---------|
| 时间为空 | 记录异常，不进入正常汇总 |
| 时间格式错误 | 尝试解析，失败则记录异常 |
| 完成时间早于添加时间 | 记录异常但保留数据 |

### 6.6 活跃主体定义

| 主体类型 | 活跃判定规则 |
|---------|------------|
| 活跃合伙人 | 在选定时间范围内，至少有1笔订单 |
| 活跃商家 | 在选定时间范围内，至少有1笔完成订单 |
| 活跃骑手 | 在选定时间范围内，至少有1笔完成订单 |

## 7. 系统架构设计

### 7.1 架构分层

```text
文件输入层
    ↓
ODS 原始层
    ↓
DWD 明细标准层
    ↓
ADS 汇总应用层
    ↓
前端看板 / 后端接口
```

### 7.2 架构说明

**文件输入层**：负责读取订单、骑手、商家、合伙人原始文件  
**ODS 层**：保留原始数据，便于审计和重导  
**DWD 层**：核心标准明细宽表，统一核心口径  
**ADS 层**：面向页面与报表的汇总表  
**前端看板层**：展示全国和单城市分析页面

### 7.3 第一阶段设计取舍

本次方案不额外增加复杂指标层、数据服务层或独立语义层，而是将核心口径统一沉淀在 DWD 的派生逻辑中，确保：

- 指标口径统一
- 开发成本可控
- 页面和接口复用同一批汇总表

### 7.4 后续扩展方向

第二阶段如指标复用场景明显增多，可进一步引入：

- 指标配置层（Metric Layer）
- 告警任务层
- 权限与脱敏层

**建议字段：**

## 9. 数据处理流程

### 9.1 文件导入流程

程序启动或手动导入时执行以下步骤：

1. 扫描 `orders/`、`riders/`、`merchants/`、`partners/` 目录
2. 识别 `xls/xlsx/csv` 文件
3. 根据文件名、文件大小进行去重
4. 读取文件并进行编码识别：
   - 优先 UTF-8
   - 失败后尝试 GBK
5. 原始数据写入 ODS
6. 将导入状态记录到 `ods_file_registry`
7. 失败文件记录日志并继续执行下一个文件

### 9.2 字段匹配策略

第一阶段采用“有限容错”策略：

| 策略 | 说明 |
|------|------|
| 标准字段名优先 | 优先按既定模板字段匹配 |
| 大小写不敏感 | 如 OrderID / orderid |
| 少量别名映射 | 如 订单号 → 订单编号 |
| 不做高度模糊匹配 | 避免误匹配带来口径错误 |

### 9.3 数据清洗流程

清洗步骤如下：

1. 时间字段统一转标准格式
2. ID 字段统一转字符串
3. 金额字段统一转数值
4. 空值与异常字符清洗
5. 行政区拆分
6. 关联合伙人、骑手、商家名单
7. 生成 DWD 明细宽表

### 9.4 业务计算流程

在 DWD 层完成以下计算：

1. `is_paid`
2. `is_completed`
3. `is_cancelled`
4. `pay_cancel_minutes`
5. `is_valid_order`
6. `hq_subsidy`
7. `partner_subsidy`
8. `is_new_rider`
9. `is_new_merchant`
10. `is_new_partner`

### 9.5 汇总生成流程

汇总步骤如下：

1. 从 DWD 生成 `ads_admin_day_metrics`
2. 从 DWD 生成 `ads_partner_day_metrics`
3. 刷新页面使用缓存（如有）
4. 写入汇总日志

### 9.6 刷新策略

第一阶段建议使用以下策略：

- 手动导入后自动重算汇总
- 或每天凌晨定时导入 + 重算
- 页面不做实时计算，统一读 ADS 表

## 10. 数据质量与异常管理

### 10.1 第一阶段质量管理原则

第一阶段不建设复杂 DQC 平台，但必须具备最基础的数据质量防线，目标是：

- 能发现明显错误
- 能记录异常来源
- 能支持人工排查

### 10.2 导入质量校验

| 校验项 | 校验内容 | 处理方式 |
|--------|---------|---------|
| 行数校验 | 文件读取行数是否正常 | 异常时记录 |
| 必填字段校验 | 订单编号、状态、时间是否缺失 | 记录异常 |
| 时间格式校验 | 时间是否可正常解析 | 失败写异常日志 |
| 金额格式校验 | 金额字段能否转数值 | 失败写异常日志 |
| 关联校验 | 订单能否关联合伙人/商家/骑手 | 失败写异常日志 |

### 10.3 异常订单类型

以下情况写入 `dqc_abnormal_orders`：

| 异常类型 | 判定规则 |
|---------|---------|
| 主键缺失 | 订单编号为空 |
| 主键重复 | 订单编号重复 |
| 时间异常 | 支付、取消、完成时间无法解析 |
| 时间倒挂 | 完成时间早于添加时间；取消时间早于支付时间 |
| 金额异常 | 金额字段非数字 |
| 关联失败 | 合伙人、骑手、商家无法关联 |
| 行政区异常 | 行政区无法拆分 |

### 10.4 异常处理原则

- 异常数据原则上不阻断整体流程
- 但主键严重异常、文件完全不可读等情况可判定导入失败
- 异常信息必须能定位到源文件和源记录

## 附录A. 核心规则清单

> 后续重建程序时必须保留以下规则，任何变更需经过业务确认。

### A.1 主键关联规则

- `帮手ID = 订单明细.配送员ID`

- 骑手/商家的所属合伙人字段优先使用合伙人ID关联

### A.2 新主体定义规则

- 新骑手：按 `入职时间`，默认观察窗口 30 天
- 新商家：按 `注册时间`，默认观察窗口 30 天
- 新合伙人：按 `成立时间/开城时间`，默认观察窗口 90 天

### A.3 有效订单定义规则

- 有效订单 = 完成订单 + 支付后取消时长 > 5 分钟的订单
- 正好 5 分钟不计入有效订单

### A.4 补贴归属规则

- `总部优惠金额` 的归属由 `营销优惠劵ID` 决定
- 营销优惠劵ID为空 → 总部补贴
- 营销优惠劵ID不为空 → 合伙人补贴
- `优惠金额` 永远记为合伙人补贴

### A.5 商家主体口径规则

- 商家保留三套口径：`商家ID / 用户ID / 商家型用户`
- 商家型用户阈值默认 20 单

### A.6 时间归属规则

- 日汇总按 `添加时间` 的日期归属
- 小时汇总按 `添加时间` 的小时归属
- 异常时间记录到异常表

### A.7 行政区拆分规则

- 分隔符优先级：`-` > `/` > 空格
- 直辖市自动补全：省名 = 市名
- 无法识别写入异常表
