# 同城配送经营分析系统项目完整口径手册 v1

## 1. 文档说明

本文档用于沉淀当前 `delivery-dashboard` 项目的实际落地口径、数据链路、指标定义、页面展示逻辑和后续优化方向。

适用对象：
- 业务负责人
- 经营分析人员
- 运营负责人
- 后续接手开发和维护的技术人员

本文档以当前代码实际实现为准，主要参考：
- `app/pipeline.py`
- `app/api.py`
- `app/models.py`
- `config/config.yaml`

## 2. 项目目标

本项目的目标是搭建一个本地可运行、可重复导入、可查询的同城配送经营分析系统。

当前首版已实现的闭环：
- 导入订单、骑手、商家、合伙人四类数据
- 对订单明细进行标准化清洗和建模
- 生成全国视角和单合伙人城市视角的经营分析结果
- 输出直营经营专项分析
- 支持本地网页看板查询

当前不在首版范围内的内容：
- 多用户权限体系
- 云端共享部署
- 在线骑手状态日志分析
- 优惠券核销全链路分析
- 多年历史大规模分布式分析

## 3. 当前技术架构

当前技术栈：
- 后端：Python + FastAPI + SQLAlchemy
- 数据底座：DuckDB
- 前端：原生 HTML + CSS + JS
- 数据组织：ODS -> 名单标准表 -> DWD -> ADS

当前默认数据库：
- `F:/delivery-dashboard-data/delivery_analysis.duckdb`

当前默认配置文件：
- `config/config.yaml`

当前项目目录：

- `logs`：导入日志
- `app`：后端和前端代码

## 4. 数据接入与主关联关系

### 4.1 订单主关联键

当前主关联规则：
- 订单明细.配送员ID = 帮手信息.帮手ID
- 订单明细.商家ID = 商户信息.商家ID
- 订单明细.合伙人ID = 合伙人信息.ID

### 4.2 区域维度来源

当前区域维度来自合伙人名单中的区域字段。

处理方式：
- 先读取 `partner_roster.region_raw`
- 再解析为 `province / city / district`
- DWD 明细优先继承合伙人名单解析后的区域

支持的区域格式包括：
- `广东省-广州市-天河区`
- `广东省/广州市/天河区`
- `广东省 广州市 天河区`
- `广东省广州市天河区`
- `北京市朝阳区`
- `广西壮族自治区南宁市青秀区`

### 4.3 订单已接入关键字段

当前订单明细已接入并入模的主要字段包括：
- 订单ID
- 合伙人ID、合伙人名称
- 商家ID、商家名称、店铺名称
- 用户ID
- 骑手ID、骑手名称
- 在职状态
- 订单状态
- 客服编号
- 下单来源
- 下单时间、支付时间、接单时间、取消时间、完成时间
- 订单价格、应付金额、实付金额
- 优惠券ID、营销优惠券ID
- 总部优惠金额、优惠金额

- 全国汇总

- 直营专项汇总

当前默认配置如下：

- 分块大小：`50000`
- 中间格式：`csv`
- `raw_payload` 默认关闭

### 6.2 业务参数

- 有效订单取消阈值：`5分钟`
- 新骑手观察窗口：`30天`
- 新商家观察窗口：`30天`
- 新合伙人观察窗口：`90天`
- 商家型用户默认阈值：`20单`

### 6.3 波动预警参数

大城市：
- 日均完成单 `>= 500`
- 变化量 `>= 100`
- 幅度 `>= 20%`

中城市：
- 日均完成单 `200-499`
- 变化量 `>= 50`
- 幅度 `>= 25%`

小城市：
- 变化量 `>= 30`
- 幅度 `>= 30%`

## 7. DWD 明细层口径定义

### 7.1 基础状态字段

`is_paid`
- 支付时间非空时为真

`is_completed`
- 完成时间非空时为真

`is_cancelled`
- 取消时间非空时为真

### 7.2 时间衍生字段

`order_date`
- 取下单时间对应日期

`order_hour`
- 取下单时间对应小时

`accept_hour`
- 取接单时间对应小时

`pay_cancel_minutes`
- 支付时间到取消时间的分钟差

`order_elapsed_minutes_to_cancel`
- 下单时间到取消时间的分钟差

`is_cross_day_order`
- 下单日期与完成日期不在同一天

### 7.3 有效订单

`is_valid_order`
- 完成订单记为有效订单
- 或支付后取消且 `支付到取消时长 > 5分钟` 的订单记为有效订单
- 正好 `5分钟` 不算有效订单
- 未支付取消不算有效订单

### 7.4 有效取消订单

`is_valid_cancel_order`
- 取消订单
- 且 `支付到取消时长 > 5分钟`

说明：
- 这就是当前直营报表中“有效取消订单”的定义

### 7.5 新主体识别

`is_new_rider_order`
- 订单日期减骑手入职日期在 `0-30天`

`is_new_merchant_order`
- 订单日期减商家注册日期在 `0-30天`

`is_new_partner_order`
- 订单日期减合伙人开城日期在 `0-90天`

说明：
- 名单缺日期时，不识别为新主体

### 7.6 客服在线

`service_online_flag`
- 下单时间小时在 `9:00-20:00` 之间

当前实际实现：
- 按小时判断
- `20:xx` 也会计入在线时段

### 7.7 超时 / 未超时取消

`is_timeout_cancel`
- 取消单
- 且下单到取消 `> 5分钟`

`is_not_timeout_cancel`
- 取消单
- 且下单到取消 `<= 5分钟`
- 或取消时长无法计算

### 7.8 无人接 / 接单后取消

`is_unaccepted_cancel`
- 取消单
- 且接单时间为空

`is_accepted_cancel`
- 取消单
- 且接单时间非空

### 7.9 无责取消

`is_rider_noliability_cancel`
- 取消单
- 且订单状态文本包含“无责”

### 7.10 券单识别

`has_coupon_order`
- `coupon_id` 非空
- 或 `marketing_coupon_id` 非空

### 7.11 全职 / 兼职识别

`employment_type`
- 在职状态包含“全职” -> `fulltime`
- 在职状态包含“兼职” -> `parttime`
- 其他 -> `unknown`

### 7.12 补贴归属

`hq_subsidy_amount`
- 当订单存在总部优惠金额
- 且 `marketing_coupon_id` 为空
- 则总部优惠金额记为总部补贴

`partner_subsidy_amount`
- 优惠金额始终计入合伙人补贴
- 若 `marketing_coupon_id` 非空，则总部优惠金额也计入合伙人补贴

实际规则可概括为：
- 无营销券：总部优惠归总部，优惠金额归合伙人
- 有营销券：总部优惠和优惠金额都归合伙人

## 8. ADS 汇总层指标定义

### 8.1 全国页指标

主要来源：
- `ads_admin_partner_metrics`

指标定义：

`总订单`
- 订单总数

`有效订单`
- `is_valid_order = true` 的订单数

`完成订单`
- `is_completed = true` 的订单数

`取消订单`
- `is_cancelled = true` 的订单数

`完成率`
- `完成订单 / 有效订单`
- 若有效订单为 0，接口层会退回用总订单兜底防止除零

`活跃合伙人数`
- 当前筛选范围内有订单的合伙人去重数

`新合伙人数`
- 当前筛选范围内 `is_new_partner = true` 的合伙人去重数

`活跃商家数`
- 完成订单商家去重数

`新商家数`
- 完成订单且为新商家的商家去重数

`活跃骑手数`
- 完成订单骑手去重数

`新骑手数`
- 完成订单且为新骑手的骑手去重数

`总部补贴`
- `hq_subsidy_amount` 汇总

`合伙人补贴`
- `partner_subsidy_amount` 汇总

### 8.2 全国区域排名

当前区域排名逻辑：
- 优先显示 `province / city / district`
- 若区域为空，则回退到 `partner_name`
- 再为空，回退到 `partner_id`

排序方式：
- 按完成单量降序

### 8.3 合伙人波动预警

预警对象：
- `ads_partner_day_metrics.completed_orders`

当前实现方式：
- 取当前筛选范围内每个合伙人的最后一天
- 取前最多 7 天的完成单量均值作为基线
- 计算：
  - `change_abs = 最新完成单 - 基线`
  - `change_pct = change_abs / 基线`
- 再按大中小城市阈值判断是否触发预警

### 8.4 城市页日汇总

主要来源：
- `ads_partner_day_metrics`

指标定义：

`完成率`
- `completed_orders / valid_orders`

`取消率`
- `cancelled_orders / total_orders`

`新骑手完成单量`
- `is_completed = true and is_new_rider_order = true`

`老骑手完成单量`
- `is_completed = true and rider_id 非空 and not is_new_rider_order`

`新商家完成单量`
- `is_completed = true and is_new_merchant_order = true`

`老商家完成单量`
- `is_completed = true and merchant_id 非空 and not is_new_merchant_order`

### 8.5 城市页小时汇总

主要来源：
- `ads_partner_hour_metrics`

指标定义：

`小时完成单量`
- 该合伙人该日该小时的完成订单数

`小时取消单量`
- 该合伙人该日该小时的取消订单数

`小时取消率`
- `cancelled_orders / total_orders`

说明：
- 这里的小时是按下单小时 `order_hour` 归类

### 8.6 商家型用户

主要来源：
- `ads_partner_user_merchant_metrics`

识别逻辑：
- 对用户完成单量做累计
- `completed_orders >= merchant_like_threshold`
- 默认阈值 `20`

### 8.7 直营取消数据

主要来源：
- `ads_direct_cancel_day_metrics`

指标定义：

`有效订单`
- `is_valid_order = true`

`有效取消订单`
- `is_valid_cancel_order = true`

`有效订单取消率`
- `有效取消订单 / 有效订单`

`客服在线超时无人接取消`
- `is_unaccepted_cancel and is_timeout_cancel and service_online_flag`

`客服离线超时无人接取消`
- `is_unaccepted_cancel and is_timeout_cancel and not service_online_flag`

`无人接超时取消`
- `is_unaccepted_cancel and is_timeout_cancel`

`无人接未超时取消`
- `is_unaccepted_cancel and is_not_timeout_cancel`

`无人接取消`
- `is_unaccepted_cancel`

`接单后无责取消`
- `is_accepted_cancel and is_rider_noliability_cancel`

`未支付取消`
- `is_cancelled and not is_paid`

`客服在线超时无人接取消率`
- `客服在线超时无人接取消 / 总订单`

### 8.8 直营小时数据

主要来源：
- `ads_direct_hour_metrics`

指标定义：

`未支付订单`
- `is_cancelled and not is_paid`

`无人接取消`
- `is_unaccepted_cancel`

`接单后取消`
- `is_accepted_cancel`

`已送达`
- `is_completed`

`有效订单取消率`
- `valid_cancel_orders / valid_orders`

`接单骑手数`
- 按 `accept_hour` 维度，对接单骑手去重计数

`兼职人效`
- `兼职完成订单数 / 兼职骑手数`

`全职人效`
- `全职完成订单数 / 全职骑手数`

### 8.9 直营新骑手 / 新商家

新骑手来源：
- `ads_direct_new_rider_metrics`

指标：
- 帮手ID
- 帮手姓名
- 入职时间
- 发单量
- 完成单量

新商家来源：
- `ads_direct_new_merchant_metrics`

指标：
- 商家ID
- 商户名称
- 店铺/地址
- 注册时间
- 发单量
- 完成单量
- 完单率

### 8.10 商家对比

主要来源：
- `ads_direct_merchant_day_metrics`

对比规则：
- 若明确传入基线周期和当前周期，则按传入周期比较
- 若只传当前周期，则自动取上一等长周期作为基线

主要输出：
- 当前总订单
- 基线总订单
- 订单变化
- 当前完单率
- 基线完单率
- 当前平均单价
- 当前无人接取消单量与均价
- 当前接单后取消单量与均价
- 当前完成单量与均价

平均单价口径：
- 实际按 `amount_paid` 聚合计算

### 8.11 渠道对比

主要来源：
- `ads_direct_order_source_day_metrics`

维度：
- `order_source`

当前输出：
- 基线总订单
- 当前总订单
- 总订单变化
- 基线无人接取消单量
- 当前无人接取消单量
- 无人接取消变化

### 8.12 优惠券分析

主要来源：
- `ads_direct_coupon_metrics`

分组键：
- `marketing_coupon_id + coupon_id`

当前输出：
- 券单量
- 总部优惠金额
- 优惠金额
- 总优惠金额

当前显示标签：
- 优先显示 `marketing_coupon_id`
- 否则显示 `coupon_id`
- 都没有则显示“未识别券”

## 10. 当前已知优势

当前项目已经具备的优势：
- 订单主口径已经集中在 DWD，不是散落在前端
- ADS 已按页面使用场景拆表，查询时不扫原始明细

- 订单导入已支持中间 CSV 和按月增量重建

## 11. 当前最值得优化的方向

### 11.1 优先级最高：编码与字段映射治理

当前仍然存在一些编码污染问题，尤其是在：
- 订单字段映射别名
- 部分诊断文案
- 个别默认提示文本

建议：
- 统一将后端和前端源码全部整理为 UTF-8
- 清理字段别名中的乱码
- 做一份字段字典表

### 11.2 优先级高：订单状态标准化

当前部分逻辑仍依赖订单状态文本包含判断。

风险：
- 不同导出模板可能状态文案不同
- 未来新增城市可能字段不一致

建议：
- 增加标准状态映射层
- 将完成、取消、无责等状态统一转换后再入 DWD

### 11.3 优先级高：正式口径手册版本化

建议把以下内容版本化管理：
- 有效订单定义
- 有效取消定义
- 新主体窗口
- 客服在线时间范围
- 商家型用户阈值
- 补贴归属规则

建议增加：
- 口径版本号
- 生效日期
- 修改记录

### 11.4 优先级高：取消分析进一步细化

当前直营取消已可用，但还可以补：
- 接单前取消率
- 接单后取消率
- 有效取消占全部取消比例
- 超时无人接占全部取消比例
- 按渠道的取消率
- 按商家的取消率

### 11.5 优先级中高：骑手与商家生命周期分析

建议增加：
- 首单商家
- 沉默商家
- 回流商家
- 连续 7 天未发单商家
- 新骑手 7 日留存
- 新骑手首周完成单分布
- 稳定活跃骑手数

### 11.6 优先级中：性能继续优化

建议优化方向：
- 更严格复用 `orders_stage` 中间文件
- 更细粒度 ADS 重建，只重建受影响月份和受影响合伙人
- `meta` 和全国首页结果增加缓存
- 页面首屏先加载摘要，再异步加载明细

### 11.7 优先级中：页面导出能力

建议增加：
- 当前筛选结果导出 Excel
- 当前图表导出 PNG
- 页面自动生成日报摘要

### 11.8 优先级中：区域与全国分析回钻

建议增强：
- 省 -> 市 -> 区逐级回钻
- 各层都展示完成单、取消率、补贴、波动

## 12. 当前需要业务继续确认的口径点

以下口径建议后续明确成制度化标准：
- 全国页完成率是否永久按 `完成单 / 有效单`
- 城市页取消率是否保留 `取消单 / 总订单`
- 客服在线是否固定包含 `20:00-20:59`
- 无责取消是否永久按状态文本包含“无责”
- 商家型用户阈值是否允许按城市配置
- 渠道分析是否要加入完成率和取消率变化
- 优惠券分析是否要拆总部券、合伙人券、营销券

## 13. 结论

当前项目已经从“方案阶段”进入“可运行、可查询、口径基本成型”的阶段。

现阶段最值得继续投入的方向，不是继续堆更多图表，而是先把以下四件事做扎实：
- 编码和字段映射清理
- 正式口径手册版本化
- 订单状态标准化
- 骑手 / 商家生命周期分析

做到这一步后，这套系统会从“能用的本地分析工具”升级成“稳定可复用的经营分析底座”。
