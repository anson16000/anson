# 同城配送经营分析系统方案

## 1. 文档目标
本文档用于沉淀同城配送经营分析系统的完整建设方案，供后续在飞书知识库中长期保存、复用和迭代。

系统目标：
- 自动读取订单明细、骑手名单、商家名单、合伙人名单
- 自动导入数据库并完成清洗、建模、汇总
- 自动生成本地网页看板
- 支持全国行政区分析和单个合伙人城市分析
- 支持后续按本文档重新生成程序

---

## 2. 系统总体目标
将当前依赖 Excel 手工分析的模式，升级为一个可持续运行的本地经营分析系统。

系统重点解决以下问题：
- 哪些行政区、哪些合伙人经营表现更好或更差
- 新合伙人开城后完成得怎么样
- 哪些合伙人单量出现大幅波动
- 单个城市每天、每小时的完成与取消情况如何
- 新骑手、新商家是否形成有效供给
- 总部补贴与合伙人补贴分别产生了什么效果

## 4. 主键与字段映射规则

### 4.1 订单明细主字段
订单明细表至少使用以下字段：
- `订单编号`
- `合伙人ID`
- `商家ID`
- `用户ID`
- `配送员ID`
- `订单状态`
- `添加时间`
- `支付时间`
- `取消时间`
- `完成时间`
- `总部优惠金额`
- `营销优惠劵ID`
- `优惠金额`

### 4.2 骑手字段映射
来源：`帮手信息.xlsx`

字段映射：
- `帮手ID = 配送员ID`
- `帮手姓名 -> rider_name`
- `入职时间 -> hire_date`
- `状态 -> rider_status`
- `所属合伙人 -> rider_partner_name`
- `区域 -> rider_region`

规则：
- 骑手主键使用 `帮手ID`
- 新骑手只按 `入职时间` 判断

### 4.3 商家字段映射
来源：`商户信息.xlsx`

字段映射：
- `商家ID -> merchant_id`
- `商家名称 -> merchant_name`
- `所属合伙人 -> merchant_partner_name`
- `所属区域 -> merchant_region`
- `注册时间 -> merchant_register_date`
- `状态 -> merchant_status`

规则：
- 商家主键使用 `商家ID`
- 新商家只按 `注册时间` 判断

### 4.4 合伙人字段映射
来源：`2026.4.1合伙人信息.xlsx`

字段映射：
- `ID = 订单明细.合伙人ID`
- `合伙人公司名 -> partner_name`
- `成立时间 -> partner_open_date`
- `合伙人区域 -> partner_region_raw`
- `状态 -> partner_status`

规则：
- 合伙人主键使用 `ID`
- 新合伙人按 `成立时间` 作为 `开城时间`
- `合伙人区域` 用于全国行政区分析

## 5. 核心业务口径

### 5.1 有效订单
有效订单定义：

```text
有效订单 = 完成订单 + 支付5分钟后取消的订单
```

判定规则：
- `完成订单`：订单状态为已送达
- `支付5分钟后取消订单`：存在支付时间和取消时间，且取消时间减支付时间大于 5 分钟
- 不计入有效订单：
  - 未支付取消
  - 支付后 5 分钟内取消

### 5.2 新主体定义
- 新骑手：按骑手名单 `入职时间`
- 新商家：按商家名单 `注册时间`
- 新合伙人：按合伙人名单 `成立时间/开城时间`

### 5.3 补贴归属规则
- `总部优惠金额` 且 `营销优惠劵ID` 为空：记为 `总部补贴`
- `总部优惠金额` 且 `营销优惠劵ID` 不为空：记为 `合伙人补贴`
- `优惠金额`：一律记为 `合伙人补贴`

### 5.4 商家主体口径
商家保留三套分析口径：
- `商家ID口径`
- `用户ID口径`
- `商家型用户口径`

商家型用户定义：
- 在当前筛选范围内，`用户ID订单量 >= 20 单`，视为商家型用户

## 7. 数据处理流程

### 7.1 文件导入流程
程序启动导入时执行以下步骤：
1. 扫描 `orders/`、`riders/`、`merchants/`、`partners/` 四个目录
2. 识别全部 `xls/xlsx/csv`
3. 根据文件名、文件大小、文件哈希进行去重
4. 将原始数据写入 ODS 表
5. 将导入结果写入日志表和日志文件

### 7.2 数据清洗流程
清洗时执行以下步骤：
1. 标准化时间字段
2. 统一空值、`-`、`0`
3. 重复字段重命名
4. 金额字段转数值
5. ID 字段统一为字符串
6. 行政区字段拆分为 `省 / 市 / 区县`
7. 关联骑手、商家、合伙人维表

### 7.3 业务计算流程
业务计算时执行以下步骤：
1. 计算支付后取消分钟数
2. 判断是否为有效订单
3. 计算总部补贴金额
4. 计算合伙人补贴金额
5. 判断是否为新骑手订单
6. 判断是否为新商家订单
7. 判断是否为新合伙人城市订单
8. 判断是否为商家型用户订单

### 7.4 汇总输出流程
汇总时执行以下步骤：
1. 生成全国行政区日汇总
2. 生成行政区-合伙人汇总
3. 生成合伙人城市日汇总
4. 生成合伙人城市小时汇总
5. 生成骑手日汇总
6. 生成商家日汇总
7. 生成商家型用户汇总

- 是否只看新商家
- 是否只看新骑手

核心指标：
- 总订单
- 有效订单
- 完成订单
- 取消订单
- 活跃合伙人数
- 新合伙人数
- 活跃商家数
- 新商家数
- 活跃骑手数
- 新骑手数
- 总部补贴
- 合伙人补贴

核心模块：
- 行政区排名
- 新合伙人表现
- 合伙人波动预警
- 区域诊断

### 8.2 单个合伙人城市页
用途：
- 看单个城市经营质量
- 看每天、每小时的完成和取消情况
- 看新骑手、新商家表现

筛选项：
- 日期范围
- 合伙人
- 商家型用户阈值

核心模块：
- 城市总览
- 每日完成/取消趋势
- 小时热力图
- 骑手分析
- 商家分析
- 经营诊断

### 8.3 小时展示方式建议
单城市小时数据主展示使用图形化热力图：
- `完成单量热力图`
- `取消单量热力图`
- `取消率热力图`

展示建议：
- 横轴：小时 `0-23`
- 纵轴：日期
- 颜色越深表示值越高

辅助图表：
- 每小时完成单量柱状图
- 每小时取消率折线图

## 9. 重点分析内容

### 9.1 新合伙人表现
建议输出：
- 新合伙人数
- 新合伙人完成单量
- 新合伙人有效订单量
- 新合伙人完成率
- 新合伙人日均完成单量
- 新老合伙人对比

### 9.2 合伙人波动预警
建议识别：
- 完成单量异常上涨合伙人
- 完成单量异常下滑合伙人

第一版推荐阈值：
- 绝对变化量 `>= 30 单`
- 变化幅度 `>= 30%`

同时展示辅助原因：
- 新商家变化
- 新骑手变化
- 活跃商家变化
- 活跃骑手变化
- 取消率变化
- 补贴变化

### 9.3 新骑手分析
建议输出：
- 新骑手数
- 新骑手完成单量
- 老骑手完成单量
- 新骑手完成单量占比
- 新骑手完成率
- 新骑手人均完成单量

### 9.4 新商家分析
建议输出：
- 新商家数
- 新商家完成单量
- 老商家完成单量
- 新商家完成单量占比
- 新商家完成率
- 新商家单均补贴

### 9.5 商家主体识别分析
同时输出三套主体口径：
- 商家ID口径
- 用户ID口径
- 商家型用户口径

用于识别：
- 用户实际为商家的城市
- 商家主体识别偏差较大的城市

- 文件放入后可自动识别和导入
- 重复文件不会重复导入
- 骑手、商家、合伙人名单能正确关联订单
- 新骑手、新商家、新合伙人口径正确
- 有效订单口径正确
- 补贴归属口径正确
- 全国页与单城市页筛选正确
- 热力图与汇总结果一致
- 波动预警和经营诊断能够正常输出

## 13. 后续重建程序时必须保留的规则
- `帮手ID = 配送员ID`
- `合伙人信息.ID = 订单明细.合伙人ID`
- 新骑手只按 `入职时间`
- 新商家只按 `注册时间`
- 新合伙人只按 `成立时间/开城时间`
- `有效订单 = 完成订单 + 支付5分钟后取消订单`
- `总部优惠金额` 的归属由 `营销优惠劵ID` 决定
- `优惠金额` 永远记为合伙人补贴
- 商家同时保留三套口径：`商家ID / 用户ID / 商家型用户`
