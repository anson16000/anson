# Power BI 诊断优先报表设计方案

版本：v1.0  
日期：2026-05-13  
适用场景：使用 `Python + DuckDB + Parquet + Power BI` 重构同城配送经营分析报表

## 1. 设计目标

这套 Power BI 报表不建议简单复刻网页看板，而是改成“诊断优先”的经营分析路径：

```text
先看总体是否正常
-> 找出异常城市 / 合伙人
-> 判断异常原因
-> 下钻到单个城市
-> 看时段、骑手、商家、订单来源
-> 输出提升动作建议
```

这样做的好处是：

- 管理者先看到结论，不需要在很多明细表里找问题。
- 城市负责人能快速判断问题是订单不足、骑手不足、人效不足、商家不足，还是履约异常。
- 数据分析人员可以继续下钻到骑手、商家、订单来源和时段。
- 后续可以把异常原因和改善建议做成固定规则，减少人工判断。

## 2. 推荐页面结构

最终建议做 7 页：

| 页码 | 页面 | 主要回答的问题 |
| --- | --- | --- |
| 第 1 页 | 经营总览 | 整体经营是否正常？本月表现比上月同期好还是差？ |
| 第 2 页 | 异常诊断 | 哪些城市 / 合伙人异常？异常原因是什么？ |
| 第 3 页 | 城市经营 | 单个城市的问题是订单、人员、人效、商家还是补贴？ |
| 第 4 页 | 时段热力与履约 | 哪些时段接不住？高峰骑手够不够？履约有没有问题？ |
| 第 5 页 | 主体分析 | 骑手、商家、订单来源、新主体、达标情况怎么样？ |
| 第 6 页 | 提升建议 | 针对异常应该做什么动作？优先处理谁？ |
| 第 7 页 | 数据质量与刷新状态 | 数据是否最新？匹配是否完整？报表是否可信？ |

第 7 页可以在正式展示时隐藏，但建议保留，方便排查数据问题。

## 3. 第 1 页：经营总览

### 3.1 页面定位

经营总览是“老板页”和“日报首页”。  
它不负责解释所有细节，只负责告诉用户：

- 当前总盘子怎么样？
- 本月截至当前日期，比上月同期是增长还是下降？
- 哪些核心指标异常？
- 需要去哪个页面继续看原因？

### 3.2 顶部筛选

建议保留：

- 日期范围
- 省
- 市
- 区县
- 合伙人
- 是否新合伙人

日期筛选建议默认使用“本月截至最新数据日期”。

### 3.3 核心 KPI 卡片

建议顶部放 10 个以内，不要铺太多：

| KPI | 含义 |
| --- | --- |
| 总订单 | 当前筛选范围内全部订单 |
| 完成订单 | 最核心的业务产出 |
| 有效订单 | 完成订单 + 有效取消订单 |
| 完成率 | 完成订单 / 总订单 |
| 取消率 | 取消订单 / 总订单 |
| 活跃骑手数 | 有完成订单的骑手数 |
| 活跃商家数 | 有完成订单的商家数 |
| 总部补贴 | 总部承担的补贴 |
| 合伙人补贴 | 合伙人承担的补贴 |
| 经营利润 | 合伙人收入总额 - 合伙人补贴 |

### 3.4 核心图表

建议图表：

| 图表 | 用途 |
| --- | --- |
| 每日完成单量趋势 | 看整体单量走势 |
| 本月截至当前日期 vs 上月同期 | 看当前进度是否落后 |
| 完成率 / 取消率趋势 | 看履约质量是否稳定 |
| 区域完成单量排名 | 找高贡献区域和低贡献区域 |
| 异常合伙人数量卡片 | 快速跳转到异常诊断 |

### 3.5 建议新增指标

建议在 Power BI 中增加这些度量值：

| 指标 | 用途 |
| --- | --- |
| 上月同期完成单量 | 用来做同口径进度对比 |
| 同期完成单量差值 | 本期完成单量 - 上月同期完成单量 |
| 同期完成单量增长率 | 判断增长或下滑幅度 |
| 异常合伙人数 | 快速判断风险面大小 |
| 风险合伙人数 | 判断需要重点跟进的对象数量 |

## 4. 第 2 页：异常诊断

### 4.1 页面定位

异常诊断页回答两个问题：

1. 哪些城市 / 合伙人有问题？
2. 问题大概率是什么原因？

这一页是整套报表的核心，不只是展示排名，而是帮助判断经营动作。

### 4.2 异常类型

第一版建议先用规则判断，保证可解释：

| 异常类型 | 判断思路 |
| --- | --- |
| 单量下滑 | 本期完成单量低于上月同期，且下降幅度超过阈值 |
| 完成率低 | 完成率低于目标值 |
| 取消率高 | 取消率高于目标值 |
| 骑手不足 | 活跃骑手数下降，或高峰接单骑手不足 |
| 人效低 | 完成订单 / 接单骑手数低于目标 |
| 商家活跃不足 | 活跃商家数下降，或商家贡献下降 |
| 高峰承接不足 | 高峰时段订单多但接单骑手少、人效低或取消高 |
| 订单来源异常 | 某个订单来源占比异常下降或过度集中 |

### 4.3 推荐图表

| 图表 | 用途 |
| --- | --- |
| 异常合伙人清单 | 明确优先处理对象 |
| 异常类型矩阵 | 看每个合伙人命中了哪些问题 |
| 完成单量变化 vs 活跃骑手变化散点图 | 判断是不是人员问题 |
| 完成单量变化 vs 活跃商家变化散点图 | 判断是不是商家问题 |
| 风险优先级排行榜 | 决定先处理谁 |

### 4.4 推荐字段

异常清单建议至少包含：

- 合伙人ID
- 合伙人名称
- 省
- 市
- 区县
- 完成订单
- 上月同期完成订单
- 完成订单差值
- 完成订单变化率
- 完成率
- 取消率
- 活跃骑手数
- 活跃商家数
- 人效
- 异常类型
- 风险等级
- 建议动作

### 4.5 建议新增指标

| 指标 | 用途 |
| --- | --- |
| 异常原因标签 | 让用户不用自己判断原因 |
| 风险等级 | 高 / 中 / 低 |
| 优先级分数 | 用于排序 |
| 连续异常天数 | 判断是否持续恶化 |
| 是否连续异常 | 用于筛选长期问题城市 |

## 5. 第 3 页：城市经营

### 5.1 页面定位

城市经营页用于分析单个城市或合伙人的经营情况。  
它要回答：

- 单量有没有下降？
- 是订单不足，还是人员不足？
- 骑手够不够？
- 商家活跃够不够？
- 补贴有没有效率？
- 经营利润是否正常？

### 5.2 推荐展示

| 模块 | 内容 |
| --- | --- |
| 城市摘要 | 完成订单、完成率、取消率、活跃骑手、活跃商家、补贴、利润 |
| 每日单量 | 当前周期每日完成订单 |
| 上月同期 | 同日期范围对比上月同期 |
| 骑手趋势 | 活跃骑手数、骑手人效 |
| 商家趋势 | 活跃商家数、商家单均产出 |
| 收益补贴 | 收入、补贴、利润、单均补贴 |
| 问题摘要 | 当前城市主要异常原因 |

### 5.3 判断逻辑

| 现象 | 可能原因 |
| --- | --- |
| 单量下降，骑手没下降 | 订单来源或商家活跃问题 |
| 单量下降，骑手也下降 | 人员供给问题 |
| 单量没降，取消率升高 | 履约质量问题 |
| 活跃商家下降 | 商家流失或激活不足 |
| 补贴升高，单量没升 | 补贴效率偏低 |
| 高峰时段取消高 | 高峰承接不足 |

### 5.4 建议新增指标

| 指标 | 用途 |
| --- | --- |
| 单均补贴 | 总补贴 / 完成订单 |
| 补贴拉动效率 | 单量增长 / 补贴增长 |
| 活跃骑手单均产出 | 完成订单 / 活跃骑手数 |
| 活跃商家单均产出 | 完成订单 / 活跃商家数 |
| 经营利润率 | 经营利润 / 合伙人收入总额 |

## 6. 第 4 页：时段热力与履约

### 6.1 页面定位

时段热力与履约页用来看“哪个小时出问题”。  
它要回答：

- 哪些小时单量最高？
- 高峰骑手够不够？
- 全职和兼职人效差异大不大？
- 哪些小时取消率高？
- 哪些小时 SLA 超时严重？

### 6.2 推荐展示

| 模块 | 内容 |
| --- | --- |
| 小时单量热力图 | 日期 x 小时的完成单量 |
| 小时接单骑手数 | 总接单骑手、全职接单骑手、兼职接单骑手 |
| 小时人效 | 总人效、全职人效、兼职人效 |
| 小时取消率 | 找取消高的时段 |
| SLA / 超时 | 找履约异常时段 |
| 高峰异常清单 | 输出需要排班优化的时段 |

### 6.3 需要补充的数据

当前小时汇总已经适合看接单骑手数和人效。  
如果要把履约做完整，建议后续补充：

- SLA 超时订单数
- SLA 超时率
- 平均履约时长
- 平均接单时长
- 平均配送时长
- 高峰时段标记
- 高峰接单骑手数
- 高峰人效

## 7. 第 5 页：主体分析

### 7.1 页面定位

主体分析页看骑手、商家、订单来源和用户主体结构。  
它要回答：

- 骑手是否足够？
- 骑手有没有达标？
- 商家是否活跃？
- 订单来源是否健康？
- 新骑手、新商家有没有贡献？
- 是否存在商家型用户？

### 7.2 推荐展示顺序

建议顺序：

1. 主体摘要
2. 订单来源分析
3. 商家名单
4. 骑手名单
5. 骑手达标分析
6. 骑手提成明细
7. 新骑手 / 新商家贡献趋势
8. 用户主体识别

### 7.3 骑手侧指标

| 指标 | 用途 |
| --- | --- |
| 骑手ID | 唯一识别骑手 |
| 骑手姓名 | 方便业务识别 |
| 入职日期 | 判断新老骑手 |
| 完成总单量 | 当前周期贡献 |
| 每日完成单量 | 看稳定性 |
| 达标天数 | 看有多少天达到要求 |
| 是否达标 | 达标显示 1，不达标显示 0 |
| 骑手提成 | 看骑手收入 |
| 骑手单均提成 | 看收入结构 |

### 7.4 商家侧指标

| 指标 | 用途 |
| --- | --- |
| 商家ID | 唯一识别商家 |
| 商户名称 | 业务展示主名称 |
| 注册日期 | 判断新老商家 |
| 完成订单 | 当前周期贡献 |
| 活跃天数 | 判断是否稳定出单 |
| 是否新商家 | 看新商家贡献 |
| 是否商家型用户 | 识别疑似商家型用户 |

### 7.5 订单来源指标

| 指标 | 用途 |
| --- | --- |
| 订单来源 | 区分不同来源 |
| 总订单 | 来源规模 |
| 完成订单 | 来源有效贡献 |
| 有效订单 | 有效口径贡献 |
| 取消订单 | 来源质量 |
| 来源占比 | 判断来源结构 |
| 来源变化率 | 判断来源是否异常波动 |

### 7.6 建议新增指标

| 指标 | 用途 |
| --- | --- |
| Top 商家贡献率 | 看是否过度依赖少数商家 |
| 长尾商家数量 | 看商家结构是否健康 |
| 低活跃商家数量 | 找激活对象 |
| 新骑手 7 / 14 / 30 天贡献 | 看新骑手启动质量 |
| 新商家 7 / 14 / 30 天贡献 | 看新商家启动质量 |
| 骑手达标率 | 达标骑手数 / 活跃骑手数 |

## 8. 第 6 页：提升建议

### 8.1 页面定位

提升建议页把异常转换成动作。  
它不是再展示一堆指标，而是告诉用户：

- 谁最需要处理？
- 为什么需要处理？
- 应该做什么？
- 优先级是什么？
- 处理后有没有改善？

### 8.2 建议动作分类

| 异常原因 | 建议动作 |
| --- | --- |
| 骑手不足 | 补骑手、召回沉默骑手、调整排班 |
| 高峰承接不足 | 增加高峰班次、安排兼职补峰 |
| 人效低 | 优化骑手分配、清理低效时段 |
| 商家活跃不足 | 激活低活跃商家、重点维护头部商家 |
| 订单来源下降 | 排查来源渠道、恢复活动或入口 |
| 取消率高 | 排查取消原因、优化接单和履约 |
| 补贴效率低 | 调整补贴策略，避免无效补贴 |

### 8.3 建议新增手工维护表

Power BI 可以接一张 Excel 或 CSV 作为动作跟进表：

| 字段 | 用途 |
| --- | --- |
| 日期 | 建议生成日期 |
| 合伙人ID | 关联合伙人 |
| 合伙人名称 | 展示名称 |
| 问题类型 | 单量 / 骑手 / 商家 / 履约 / 补贴 |
| 建议动作 | 具体动作 |
| 责任人 | 谁负责 |
| 状态 | 未开始 / 处理中 / 已完成 |
| 跟进备注 | 人工补充 |
| 预计影响 | 预计提升订单或降低取消 |

这张表能让报表从“看数据”变成“管动作”。

## 9. 第 7 页：数据质量与刷新状态

### 9.1 页面定位

这页用于确认数据是否可信。  
建议保留给自己或数据维护人员使用。

### 9.2 推荐展示

| 指标 | 用途 |
| --- | --- |
| 最新数据日期 | 判断数据是否更新 |
| 最新导出时间 | 判断 Parquet 是否刷新 |
| 数据版本 | 对齐程序导出版本 |
| 订单明细行数 | 判断原始数据是否完整 |
| DWD 明细行数 | 判断标准明细是否生成 |
| ADS 汇总行数 | 判断汇总是否生成 |
| 缺失骑手姓名数量 | 排查骑手匹配 |
| 缺失商户名称数量 | 排查商家匹配 |
| 未匹配合伙人订单数 | 排查合伙人归属 |
| 异常订单数 | 排查脏数据 |

## 10. Power BI 数据模型建议

### 10.1 推荐关系

| 从表 | 字段 | 到表 | 字段 | 关系 |
| --- | --- | --- | --- | --- |
| 合伙人日汇总 | 合伙人ID | 合伙人维度 | 合伙人ID | 多对一 |
| 小时运力汇总 | 合伙人ID | 合伙人维度 | 合伙人ID | 多对一 |
| 订单来源汇总 | 合伙人ID | 合伙人维度 | 合伙人ID | 多对一 |
| 骑手日汇总 | 骑手ID | 骑手维度 | 骑手ID | 多对一 |
| 商家日汇总 | 商家ID | 商家维度 | 商家ID | 多对一 |
| 各事实表 | 日期 | 日期维度 | 日期 | 多对一 |

### 10.2 不建议激活的关系

这些字段不要直接作为主活动关系连接日期维度：

- 合伙人维度的开通日期
- 骑手维度的入职日期
- 商家维度的注册日期

原因是它们是“主体生命周期日期”，不是订单经营日期。  
如果同时和订单日期混用，Power BI 容易出现筛选混乱。

## 11. 建议 DAX 度量值

### 11.1 基础指标

```DAX
完成单量 = SUM('合伙人日汇总'[完成订单])

总订单 = SUM('合伙人日汇总'[总订单])

有效订单 = SUM('合伙人日汇总'[有效订单])

完成率 = DIVIDE([完成单量], [总订单])

取消率 = DIVIDE(SUM('合伙人日汇总'[取消订单]), [总订单])

经营利润 =
SUM('合伙人日汇总'[合伙人收入总额])
- SUM('合伙人日汇总'[合伙人补贴])

单均补贴 =
DIVIDE(
    SUM('合伙人日汇总'[总部补贴]) + SUM('合伙人日汇总'[合伙人补贴]),
    [完成单量]
)
```

### 11.2 同期对比

```DAX
上月同期完成单量 =
CALCULATE(
    [完成单量],
    DATEADD('日期维度'[日期], -1, MONTH)
)

完成单量同期差值 =
[完成单量] - [上月同期完成单量]

完成单量同期增长率 =
DIVIDE([完成单量同期差值], [上月同期完成单量])
```

### 11.3 人效

```DAX
人效 =
DIVIDE(
    SUM('小时运力汇总'[完成订单]),
    SUM('小时运力汇总'[接单骑手数])
)

全职人效 =
DIVIDE(
    SUM('小时运力汇总'[全职完成订单]),
    SUM('小时运力汇总'[全职接单骑手数])
)

兼职人效 =
DIVIDE(
    SUM('小时运力汇总'[兼职完成订单]),
    SUM('小时运力汇总'[兼职接单骑手数])
)
```

## 12. 优先补充的数据

第一优先级：

- SLA 超时订单数
- SLA 超时率
- 平均履约时长
- 高峰时段标记
- 高峰接单骑手数
- 骑手达标天数
- 骑手达标率
- 异常原因标签
- 建议动作表

第二优先级：

- Top 商家贡献率
- 长尾商家数量
- 低活跃商家数量
- 订单来源变化率
- 新骑手 7 / 14 / 30 天贡献
- 新商家 7 / 14 / 30 天贡献

第三优先级：

- 天气
- 节假日
- 活动日期
- 商圈类型
- 区域人口或消费标签

## 13. 实施顺序

建议按这个顺序做，不要一次性全堆上去：

1. 先搭数据模型：日期维度、合伙人维度、合伙人日汇总、小时运力汇总。
2. 先完成第 1 页经营总览。
3. 再做第 2 页异常诊断。
4. 再做第 3 页城市经营。
5. 再补第 4 页时段热力与履约。
6. 再补第 5 页主体分析。
7. 最后做第 6 页提升建议和第 7 页数据质量。

## 14. 验收标准

| 验收项 | 标准 |
| --- | --- |
| 数据连接 | Power BI 能读取 Parquet 文件夹 |
| 中文字段 | 字段面板显示中文字段名 |
| 经营总览 | 总订单、完成订单、有效订单与网页 / 导出结果一致 |
| 同期对比 | 本月截至当前日期能正确对比上月相同日期范围 |
| 异常诊断 | 能列出异常合伙人和异常原因 |
| 城市经营 | 选择单个合伙人后能看到单量、人员、人效、商家变化 |
| 时段履约 | 能看到小时单量、全职 / 兼职接单骑手数和人效 |
| 主体分析 | 能看到骑手、商家、订单来源、新主体和达标情况 |
| 提升建议 | 异常能对应到建议动作 |
| 数据质量 | 能看到最新数据日期、导出时间和数据版本 |

## 15. 一句话建议

这套 Power BI 不要做成“数据大杂烩”，要做成“经营诊断工具”：

```text
先告诉你哪里有问题，
再告诉你为什么有问题，
最后告诉你应该怎么改。
```
